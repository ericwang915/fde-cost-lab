⏱️ Latency Estimator (TTFT / TPOT / E2E)

Customers always ask "how fast?" Prefill (TTFT) is compute-bound; decode (per-token) is memory-bandwidth-bound — single-stream decode speed ≈ HBM bandwidth ÷ model bytes. Batching raises aggregate throughput, not single-request latency.

decode tok/s ≈ HBM_BW ÷ model_bytes  ·  TPOT = 1 ÷ decode_tok/s  ·  TTFT ≈ 2·N·prompt ÷ (FLOPS·MFU)  ·  E2E = TTFT + out × TPOT

📈 Capacity & GPU Sizing (Little's Law)

From target traffic to "how many GPUs." In-flight concurrency follows Little's Law = QPS × latency; GPU count follows the per-GPU token throughput and a utilization headroom.

concurrency = QPS × latency  ·  QPS/GPU = agg_tok/s ÷ out_tokens  ·  GPUs = ⌈ QPS ÷ (QPS/GPU) ÷ util ⌉

⚖️ Break-even: Self-host vs Managed API

Owning GPUs is a fixed monthly cost; the API is per-token. There's a crossover volume above which self-hosting wins — and below which you're paying for idle silicon.

self-host $/mo = #GPU × $/hr × 24 × 30  ·  ceiling = #GPU × tok/s × 2.59M/mo  ·  break-even tok/s per GPU = $/hr × 1e6 ÷ (3600 × API $/1M)

🧮 KV-Cache VRAM Calculator

What dictates which GPU you buy is often the KV-cache, not the parameter count. It grows linearly with both concurrency and context length.

KV / req = 2 × layers × kv_heads × head_dim × seq_len × bytes  →  total VRAM = weights + KV/req × concurrency

🎓 Training / Fine-tune Cost

The canonical 6 × N × D rule: training a dense model takes ~6 FLOPs per parameter per token (forward + backward). Memory for full fine-tuning is dominated by optimizer state, not weights.

FLOPs = 6 × N × D  ·  time = FLOPs ÷ (#GPU × TFLOPS × MFU)  ·  full-FT memory ≈ ~16–20 bytes/param (weights+grads+Adam)  ·  LoRA ≈ weights only

🔎 RAG / Vector-DB Sizing

Two costs people forget: the vector store (RAM/disk) and the per-query context overhead — every retrieved chunk is billed as input tokens on every call.

store = vectors × dim × bytes × overhead  ·  embed cost = vectors × chunk_tokens × price ÷ 1e6  ·  added input/query = top-k × chunk_tokens
